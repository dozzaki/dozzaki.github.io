Introdução

Atualmente, a maior mudança no desenvolvimento de software é a frequencia de implantações. As equipes de produto implantam versões para produção mais cedo (e com mais frequencia). Os ciclos de lançamento de meses ou anos estão se tornando raros, especialmente entre aqueles que desenvolvem produtos e serviços baseado em software.

Os benefícios de ciclos de implantação mais curtos são claros, time to market reduzido, os clientes obtêm valor do produto em menos tempo e o feedback do cliente também flui de volta para a equipe do produto mais rápido, o que significa que a equipe pode iterar os recursos e corrgir problemas mais rapidamente.

No entenato, essa mudança também cria novos desafios para operações. Com implantações mais frequentes, é mais provável que o código implantado possa afetar negativamente a confiabilidade do serviço ou a experiência do cliente. É por isso que é importante desenvolver estratégias para implantar códigos que minimizem os riscos para o produto e os clientes.

O desenvolvimento de software ágil quebrou alguns dos silos entre análise, teste e desenvolvimento de requisitos. Implantação, operações e manutenção são outras atividades que sofreram uma separação semelhante do restante do processo de desenvolvimento de software. O movimento DevOps visa remover esses silos e incentivar a colaboração entre desenvolvimento e operações.

“Combinação de cultura, ferramentas e práticas de desenvolvimento de software e operações de sistemas, cujo principal objetivo é aumentar a colaboração entre os envolvidos no processo de entrega de software, de um modo que possamos entregar valor mais rápido e de modo mais confiável.”

DevOps tornou-se possível em grande parte devido a uma combinação de novas ferramentas operacionais e práticas estabelecidas de engenharia ágil*, mas essas não são suficientes para obter os benefícios do DevOps. Mesmo com as melhores ferramentas, o DevOps é apenas um buzzword, se você não tem a cultura certa.

As ferramentas operacionais incluem virtualização, container, cloud computing, gerenciamento de configuração entre outras. Isso geralmente é suportado por práticas de engenharia como integração contínua, desgin evolutivo e código limpo.

Agile

É uma metodologia de desenvolvimento de software com foco em iterações pequenas e incrementais gerando releases de software rápidas, contínua e com ritmo constante e sustentável, junto com o feedback do cliente ele aborda lacunas e conflitos entre os clientes e desenvolvedores.

Lean

Pensamento lean é um processo dinâmico, determinado pelo conhecimento e focalizado no cliente, para qual todos os colaboradores da empresa eliminam de forma contínua desperdícios e criam valor.

Continuous Integration

Prática de desenvolvimento de software para garantir que o código, por exemplo em uma feature branch de um repositório de código seja mesclado com a branch principal somente depois de passar por uma série de verificações de dependências, testes de unidade e integração e uma compilação bem sucedida. Se houver erros no caminho, a compilação falhará e a equipe será notificada. Os principais objetivos da integração contínua são encontrar bugs mais rápido, melhorar a qualidade, reduzir o tempo para lançar novas atualizações de software, e o mais importante dar visão a todo o time.

Continuous Delivery

A missão aqui é garantir que podemos entregar nossa aplicação a qualquer momento. Com o CD, o artefato gerado pelo CI é empacotado e está sempre pronto para ser implantada em um ou mais ambientes.

Continuous Deployment

A implantação contínua vai um passo além da entrega contínua. Com essa prática, todas as alterações que passam em todos os estágios do seu pipeline são liberados para seus clientes. Não há intervenção humana e apenas um teste com falha impedirá que uma nova alteração seja implantada em produção.

Estratégias de Deploy

Rolling

O deploy é feito gradualmente, substituindo a versão antiga. A versão antiga só é “desligada” quando a nova versão está pronta para ser executada, ou seja em algum momento dessa estratégia de deploy ambas as versões coexistem ao mesmo tempo em produção sem afetar a funcionalidade ou a experiência do usuário.

Blue-Green

O deploy blue-green consiste em ter dois ambientes idênticos, cujo na frente desses ambientes exista um load balancer que permita o direcionamento do tráfego para o ambiente desejado. Uma das maiores vantagens dessa estratégia é que o serviço vai ter praticamente zero de indisponibilidade, já que a mudança no load balancer é praticamente instantânea. Outra vantagem é a possibilidade de poder realizar os testes em produção, antes de liberar o serviço para ser utilizado pelos usuários. A desvantagem é que por essa estratégia precisar de um mirror ela irá consumir mais recursos de infraestrutura. Além disso por haver a troca física do switch pelo load balancer a session, por exemplo, será perdida e por isso seu código precisa estar preparado para esse tipo de abordagem.

Canary

Basicamente ela consiste em colocar a nova versão em produção, porém apenas para uma parcela de usuários. Essa configuração na maioria das ferramentas pode ser feita através de feature toggle, isso significa que podemos liberar, por exemplo, o novo serviço apenas para usuários do sexo feminino, ou para usuários com mais de 30 anos, etc. A grande vantagem dessa abordagem é poder ir testando a nova versão aos poucos com usuários reais e acompanhar os logs para ver se está tudo correndo bem, assim liberando a nova versão gradualmente. A grande desvantagem, é que se a aplicação não tem um grande volume de usuários, a validação no ambiente de produção pode demorar mais tempo do que o desejado.

O principal desafio da implantação do canary é criar uma maneira de rotear alguns usuários para a nova versão. Algumas ténicas: Expor os usuários internos à implantação do canary antes de permitir o acesso do usuário externo, roteamento básico com base no range de IP de origem ou liberar a aplicação em regiões geográficas específicas.

Feature Toggle

Feature Flag, também conhecida como Feature Toggle, é uma técnica que permite que mudanças de comportamento aconteçam na sua aplicação em tempo real sem a realização de nenhum deployment.

Basicamente você liga/desliga features na sua aplicação e para isso a condicional if é esencial.

Através de uma variável que determina se uma feature deve ser mostrada ou não e uma condicional para acionar os métodos corretos. Porém o objetivo é desacoplar a release de features do processo de deployment, então essa abordagem não é uma boa ideia, pois teríamos que alterar sempre essa variável.

Outra alternativa é a utlização de arquivos de configuração de forma dinâmica, onde inúmeros key-value determinam o comportamento das flags. Assim é possível alterar esses valores em qualquer momento e a aplicação mudará seu comportamento instantaneamente.

Vantenges: Entregas com mais frequencia; Minimizar riscos; Aumentar produtividade; Direcionar usuários para testes A/B.

Boas Práticas

Feature Toggles podem funcionar muito bem, mas de forma geral, é uma boa prática mantê-los de curta duração e com uma quantidade pequena. Flags de longa duração adicionam complexidade e torna-se um débito técnico. A limpeza é de suma importância para manter uma base de código limpa.

Um outro ponto de atenção é criar uma camada de abstração para o gerenciamento das flags para que seu código tenha o mínimo possível de acoplamento com esse recurso.
